# Typed Standards Specification v0.1 — Reviewer Orientation

## What this is

This is a pre-launch private circulation of the **Typed Standards Specification v0.1**. The specification is an open standard for **production-process attestation** of analytical artifacts — a cryptographically signed, content-addressed, capture-method-labeled record of *how* an artifact was produced, verifiable by a third party who does not trust the publisher.

The PDF + accompanying canonical-source GitHub URL below are being shared with a small set of named reviewers before public RFC publication. Public review will follow later when the spec is hosted at its institutional home.

**Reading time:** ~30-60 minutes for the full spec; ~5 minutes for this orientation document.

## About the project + stewardship

This is a pre-launch private circulation of the Typed Standards Specification v0.1. The project is currently maintained by Nathan Storey via the [civic-ai-tools workspace](https://github.com/npstorey/civic-ai-tools); institutional stewardship is under discussion and the specification will eventually be hosted at typedstandards.org under its new institutional home. For feedback on this draft, please reply directly via the channel that delivered this document.

## What feedback would help most

- **Substantive technical critique.** Anything in the normative sections (§8) that is wrong, underspecified, ambiguous, or that conflicts with how a downstream implementation would actually need to behave.
- **Positioning vs. adjacent standards.** The spec carries explicit disambiguation paragraphs for C2PA and W3C Verifiable Credentials in the Introduction (§5.5) and smaller disambiguations for project-name + RFC 3161 timestamping terminology in Conventions (§6.3). Feedback on whether these read clearly to someone familiar with the adjacent standard is high-value. Same goes for the in-toto / SLSA / Sigstore framing.
- **Adoption-posture concerns.** Whether the v0.1 surface is something a real implementer could pick up, what blockers a first non-reference implementation would hit, and what the spec is asking implementers to commit to that they might reasonably push back on.
- **Gaps or unclarities in the normative sections.** Section-level "I read this twice and still wasn't sure what was required" feedback is especially useful.

## What's out of scope

- **Line-edit polish.** This is a pre-public draft; copy polish will land before the public RFC. Reviewers should focus on substance, not prose.
- **Scope-expansion suggestions.** v0.1 is deliberately a minimum cohesive surface; adding things that "would be nice" is out of scope unless the addition is structurally required for adoption. New normative requirements come from named adopters per the Xanadu doctrine, not from review feedback alone.
- **Competing-standards advocacy beyond positioning.** Substantive comparison with C2PA / VC / in-toto / SLSA / Sigstore is welcome; arguments that the project should *be* one of those is out of scope (the project has reasons for not being).

## Reading roadmap

The spec is long (~1800 lines / ~50-70 PDF pages depending on rendering). The most efficient reading path:

1. **Abstract + §2 Status** (top of spec; ~2 minutes). The one-paragraph summary of what the standard does and the v0.1 status block.
2. **§5 Introduction**, especially §5.5 — the C2PA + W3C VC disambiguation paragraphs. (~10 minutes.) The C2PA paragraph is load-bearing for anyone reading with C2PA vocabulary loaded; the VC paragraph is subtler.
3. **§6.3 Disambiguated terms** — project name + RFC 3161 timestamping. (~3 minutes.) Quick orientation for the prefix-choice rationale.
4. **§7 Architecture** — the four mermaid diagrams (layered shape, envelope, verification flow, two-family taxonomy). (~10 minutes.) The diagrams carry the architectural surface area more efficiently than the prose. **For §7 specifically, view the GitHub URL below instead of the PDF** — pandoc emits the mermaid blocks as raw DSL source in the PDF, while GitHub renders mermaid as visual diagrams natively.
5. **§8 Normative specification** — the core of the spec. Skim §8.1-§8.10 for envelope mechanics; read §8.11 (Typed Claims) attentively if your interest is the claims-layer surface. (~20 minutes for skim, ~30-40 minutes for full read.)
6. **Appendix B (worked example)** — the most concrete try-it case (a typed-claim node for a noise-complaint trend in Bushwick). (~5 minutes.)
7. **§9 Conformance** — honest-status notes on what verifiers can and cannot check today (the offline-verifiability claim is named as a target, not a property). (~5 minutes.)

If you have less time: Abstract + §5.5 + §7 mermaid diagrams + Appendix B will give you a faithful picture of the surface area.

## How + when to send feedback

- **Channel.** Reply directly to the message that delivered this PDF. Per-reviewer channels are chosen by the maintainer; this orientation document does not enumerate them.
- **Format.** Specific section references help. Examples: "§8.11.3 paragraph 2 unclear — the relationship between `attestation/wasDerivedFrom/v1` and the source analysis's provenance graph could be made more explicit" or "Appendix B example doesn't cover the case where a single source analysis produces multiple typed claims."
- **Timeframe.** Rough timeframe is TBD per individual reviewer; the maintainer will coordinate scheduling out-of-band.
- **Granularity.** Section-level critique > paragraph-level critique > line-edit polish. The first two are high-value at this stage; the third lands later.
